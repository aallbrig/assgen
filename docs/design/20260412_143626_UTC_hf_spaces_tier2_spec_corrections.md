## The One Rule That Matters

**The spaces are a test harness for assgen, not the product.**
When a space breaks, it means the assgen handler or catalog needs fixing.
Fix assgen first. The space wrapper should be thin enough that fixing the
handler fixes the space automatically. Never paper over a handler bug with
a silent fallback or stub in the space wrapper.

## Correction 1: audiocraft is dead — remove from ALL Tier 2 specs

**Spec said:** T2-1 (`ambient.generate`) and T2-3 (`music.loop`) use:
```
audiocraft @ git+https://github.com/facebookresearch/audiocraft.git
```
with `MusicGen.get_pretrained(...)` from the audiocraft library.

**What will happen:** The audiocraft git install fails during Docker build
(Python 3.13 + conflicting torch/torchvision constraints). Same failure
mode as the Tier 1 audio spaces before they were fixed.

**Fix:** Use `transformers.MusicgenForConditionalGeneration` directly,
exactly as the deployed `assgen.audio.music.compose` space already does.
That space is RUNNING and proven.

**Pattern to copy** (from `spaces/assgen.audio.music.compose/app.py`):
```python
from transformers import AutoProcessor, MusicgenForConditionalGeneration
import scipy.io.wavfile, torch, numpy as np, tempfile

MODEL_ID = "facebook/musicgen-stereo-medium"  # or -large, -small
SAMPLE_RATE = 32_000
FRAME_RATE = 50

def _load():
    processor = AutoProcessor.from_pretrained(MODEL_ID)
    model = MusicgenForConditionalGeneration.from_pretrained(MODEL_ID)
    return processor, model

def generate(description, duration):
    processor, model = _load()
    model.to("cuda")
    inputs = processor(text=[description], padding=True, return_tensors="pt").to("cuda")
    tokens = model.generate(**inputs, max_new_tokens=int(duration * FRAME_RATE))
    # stereo models: tokens shape is (1, 2, samples) — transpose for scipy
    audio = tokens[0].cpu().float().numpy().T  # (samples, 2) for stereo WAV
    tmp = tempfile.NamedTemporaryFile(suffix=".wav", delete=False)
    scipy.io.wavfile.write(tmp.name, SAMPLE_RATE, audio)
    return tmp.name
```

For mono models (musicgen-medium, not stereo): `tokens[0, 0].cpu().float().numpy()` — no transpose.

**T2-1 ambient.generate:** Use `facebook/musicgen-stereo-large` via this pattern.
**T2-3 music.loop:** Use `facebook/musicgen-medium` (or stereo-medium) via this pattern.

## Correction 2: gradio version in requirements.txt blocks — ignore the line

**Spec says:** `gradio>=4.44.0` appears in every requirements.txt block.

**Reality:** Gradio is NOT in `requirements.txt` for any space. It is controlled
entirely by `sdk_version` in the README.md YAML frontmatter. The `requirements.txt`
generated by `sync_spaces.py` only contains `assgen[spaces]=={version}` plus
anything in `EXTRA_PIP`. The gradio lines in the spec are reference noise.

**Rule:** `sdk_version` in README.md **must be `"5.23.0"`** (or higher Gradio 5.x).
Never use 4.x. Never add `python_version` to the YAML.

## Correction 3: AudioGen is removed from transformers 5.x

**Spec said:** T2-1 notes reference `facebook/audiogen-medium` as viable.

**Reality:** `AudiogenForConditionalGeneration` no longer exists in transformers 5.x.
`facebook/audiogen-medium`'s `config.json` lacks a recognized `model_type`, causing
`AutoConfig.from_pretrained()` to raise `ValueError: Unrecognized model`. Do not use
audiogen-medium anywhere. It is permanently removed.

For sound effects: use `cvssp/audioldm2` via `diffusers.AudioLDM2Pipeline`.
For music/ambient: use MusicGen models via `MusicgenForConditionalGeneration`.

## Correction 4: Bark voice_preset routing

**Spec T2-2 (voice.clone) and any future Bark usage:**

`voice_preset` is a **processor** argument, not a `model.generate()` argument.
Passing it via `pipeline(text, forward_params={"voice_preset": ...})` raises:
```
ValueError: The following model_kwargs are not used by the model: ['voice_preset']
```

**Correct pattern** (from the fixed `spaces/assgen.audio.voice.tts/app.py`):
```python
from transformers import AutoProcessor, BarkModel

processor = AutoProcessor.from_pretrained("suno/bark-small")
model = BarkModel.from_pretrained("suno/bark-small", torch_dtype=torch.float16)

# voice_preset goes to the PROCESSOR, not to generate()
inputs = processor(text, voice_preset=voice_preset, return_tensors="pt")
inputs = {k: v.to(device) for k, v in inputs.items()}
audio_array = model.generate(**inputs)
sample_rate = model.generation_config.sample_rate
```

Use `suno/bark-small` (not full `suno/bark`) for ZeroGPU — full Bark is too slow
to cold-load within ZeroGPU's session window.

## Correction 5: Silent fallback stubs in handlers are an antipattern

Several existing assgen handlers have this pattern:
```python
try:
    return _run_real_inference(...)
except Exception as exc:
    logger.warning("Failed: %s", exc)
    return _stub_output(...)   # silent placeholder
```

This hides bugs. When the space runs the handler and inference fails, the user
gets garbage output (2s silence, placeholder cube, stub text) with no visible error.
The space becomes useless as a test harness.

**Rule:** Remove catch-all exception handlers from assgen handlers.
The stub for `_AVAILABLE = False` (transformers not installed) is acceptable —
that's a legitimate install-time check, not hiding a runtime failure.

Known handlers that had/have this issue:
- `audio_voice_tts.py` — catch-all removed in this session ✓
- `visual_model_create.py` — still has placeholder cube fallback (diffusers path)
- `narrative_dialogue_npc.py` — catch-all fallback to stub dialogue (borderline OK since text output is at least legible)
