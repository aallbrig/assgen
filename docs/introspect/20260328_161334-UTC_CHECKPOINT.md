## 1. Project Summary

**assgen** is an AI-driven game asset generation pipeline with a Kubernetes-style
client/server architecture. A Python CLI client (`assgen`) submits jobs over HTTP to a
FastAPI server (`assgen-server`) that dispatches work to HuggingFace model-backed handlers.

| Metric | Value |
|---|---|
| Language | Python 3.11+ |
| LOC | ~18,300 |
| Catalog job types | 82 |
| Implemented handlers | 82 (100% coverage) |
| Handlers with real ML inference | 29 (35%) |
| Handlers with algorithmic/utility logic | 53 (65%) |
| REST API endpoints | 11 |
| Compose pipelines | 7 |
| Test files | 9 |
| CI/CD workflows | 3 (ci, docs, release) |
| Docker images | 3 (server CPU, server CUDA, client) |

## 3. Current State Assessment

### 3.1 What Works

- **Full CLI command tree** (50+ commands) with `--help`, `--json`, `--yaml` output modes.
- **Server lifecycle**: start/stop/status, PID tracking, daemon mode, systemd service file.
- **Job queue**: QUEUED -> RUNNING -> COMPLETED/FAILED/CANCELLED with SQLite persistence.
- **SSE progress streaming**: `GET /jobs/{id}/events` for real-time progress.
- **File download**: `GET /jobs/{id}/files/{filename}` with path traversal protection.
- **Model management**: catalog resolution, HF Hub download with progress, device auto-detection.
- **Model validation**: allow-list enforcement + HF pipeline_tag compatibility checks.
- **Compose pipelines**: 7 multi-step workflows (NPC, weapon, prop, material, soundscape, ui-kit, environment).
- **Docker images**: CPU and CUDA server images with docker-compose orchestration.
- **CI/CD**: lint, test, Docker build on push; MkDocs deploy; release automation.
- **OpenAPI spec**: auto-exported, committed, versioned (API v1).
- **Config system**: XDG-compliant, YAML-based, env var overrides, Windows-compatible paths.
- **Auto-server**: client auto-starts a local server if no `server_url` configured.
- **Crash recovery**: stale RUNNING jobs reset to FAILED on server restart.

**Real ML Inference (29 handlers)** — These import and call actual ML models (transformers,
diffusers, audiocraft, bark, trimesh) with a graceful fallback to stub output when
dependencies are missing:

| Domain | Handlers | Key Models |
|---|---|---|
| Visual/Image Gen | 12 | SDXL, Hunyuan3D-2, TripoSR, Zero123++, Real-ESRGAN, IP-Adapter |

**Algorithmic/Utility (53 handlers)** — Pure Python, no ML dependencies. Fully functional:

## 4. Gaps Blocking MVP

Gaps are prioritized by impact on the core use case: **a developer on a laptop generates
game assets via a self-hosted GPU server on their LAN.**

### GAP-01: No End-to-End Verification on Windows Server [CRITICAL]

**Problem**: The server has never been tested running on Windows with CUDA. The systemd
service file, PID file management, and daemon mode assume Linux. The docker-compose GPU
profile uses NVIDIA Container Toolkit which requires WSL2 or native Linux Docker on Windows.

**Impact**: The primary deployment target (Windows desktop with GPU) may not work at all.

**Tasks**:
- [ ] Test `assgen-server start` natively on Windows (Python venv, no Docker)
- [ ] Test `assgen-server start --daemon` on Windows (PID file path: `%APPDATA%\assgen\`)
- [ ] Verify `device: cuda` detection works with Windows PyTorch + CUDA
- [ ] Test Docker Desktop + WSL2 + NVIDIA Container Toolkit path as alternative
- [ ] Document the chosen deployment path in `docs/server-setup.md`

### GAP-02: Network Accessibility — Server Binding [CRITICAL]

**Problem**: Server defaults to `host: 127.0.0.1` (localhost only). For the laptop-to-desktop
use case, the server must bind to `0.0.0.0` or a LAN IP. No documentation covers this.

**Impact**: Laptop client cannot reach desktop server out of the box.

**Tasks**:
- [ ] Add LAN deployment section to `docs/server-setup.md` with `host: "0.0.0.0"` example
- [ ] Add firewall note (Windows Firewall must allow port 8432 inbound)
- [ ] Consider adding `assgen server config set host 0.0.0.0` to quick start
- [ ] Test cross-machine connectivity (laptop -> desktop over LAN)

### GAP-03: No Happy-Path Smoke Test for Real Inference [HIGH]

**Problem**: CI only tests with stubs. There is no automated or documented manual procedure
to verify that a handler actually loads a model and produces a real output file. The
graceful fallback pattern means broken inference code is invisible until a user tries it.

**Impact**: Handlers may silently fail to real inference, producing only stub output, with
no signal to the developer.

**Tasks**:
- [ ] Create a `scripts/smoke_test_inference.py` that runs 3-5 core handlers end-to-end:
  - `visual.concept.generate` (SDXL — text-to-image)
  - `audio.sfx.generate` (AudioGen — text-to-audio)
  - `audio.music.compose` (MusicGen — text-to-music)
  - `scene.depth.estimate` (DPT — image-to-depth)
- [ ] Script should assert output files are valid (image has pixels, audio has samples)
- [ ] Document how to run: `python scripts/smoke_test_inference.py --device cuda`
- [ ] Add CI job that runs on `workflow_dispatch` with a self-hosted GPU runner (future)

### GAP-04: Compose Pipeline Not Tested End-to-End [HIGH]

**Problem**: The 7 compose pipelines (`npc`, `weapon`, `prop`, etc.) chain 5-12 steps.
None have an integration test. The `from_step` file-passing logic in `compose.py` is
untested — a single bug would silently break all pipelines.

**Impact**: The flagship feature (multi-step asset workflows) is unverified.

**Tasks**:
- [ ] Add `tests/test_compose_pipeline.py` with mock handlers that return known files
- [ ] Test file chaining: step N's output files arrive as step N+1's `upstream_files`
- [ ] Test error propagation: step N fails -> pipeline aborts with context
- [ ] Test timeout behavior per step

### GAP-05: `--json` Output Not Implemented Everywhere [HIGH]

**Problem**: `PERSONAS.md` explicitly calls out `--json` as the "single highest-priority
feature" for the CI/pipeline persona (Casey). The `--json` and `--yaml` flags exist in
the context module but their coverage across all commands is unverified.

**Impact**: Blocks CI/CD integration use case and machine-readable output for scripting.

**Tasks**:
- [ ] Audit all CLI commands for `--json` / `--yaml` support
- [ ] Add structured output to any commands missing it
- [ ] Add test assertions that `--json` produces valid JSON for key commands

### GAP-06: No Windows Install Documentation [MEDIUM]

**Problem**: `docs/install.md` and `docs/server-setup.md` focus on Linux/macOS. The
primary server target is a Windows desktop. No docs cover:
- Python venv setup on Windows
- CUDA toolkit installation for PyTorch
- `pip install "assgen[inference]"` on Windows with torch+CUDA
- Running as a Windows service (vs. systemd)
- Firewall configuration

**Tasks**:
- [ ] Add Windows server setup section to `docs/server-setup.md`
- [ ] Document PyTorch CUDA installation on Windows
- [ ] Document running as a background process on Windows (Task Scheduler or NSSM)
- [ ] Add Windows-specific troubleshooting (port conflicts, firewall, antivirus)

### GAP-07: No Authentication / Access Control [MEDIUM]

**Problem**: The server has zero authentication. Binding to `0.0.0.0` on a LAN exposes
the job queue, model downloads, and file access to anyone on the network.

**Impact**: Acceptable for a home LAN MVP, but risky if the network has other devices.

**Tasks**:
- [ ] Add optional API key authentication (header: `Authorization: Bearer <key>`)
- [ ] Generate key on first server start, store in `server.yaml`
- [ ] Client reads key from `client.yaml` or `ASSGEN_API_KEY` env var
- [ ] Document in server-setup.md

### GAP-08: Model Pre-Download UX [MEDIUM]

**Problem**: First job submission triggers a multi-GB model download, blocking the worker
for 10-30 minutes with no clear progress on the client side. The `assgen models install`
command exists but isn't part of any documented setup flow.

**Impact**: First-time user experience is confusing — job appears stuck.

**Tasks**:
- [ ] Add "pre-download recommended models" step to quickstart docs
- [ ] Create `assgen models install --recommended` that downloads the 5 most-used models
- [ ] Improve client-side messaging when a job is blocked on model download
- [ ] Add estimated download size to `assgen models list` output

### GAP-09: Handler Fallback Is Silent [MEDIUM]

**Problem**: When ML dependencies are missing, handlers fall back to stub output without
warning the user. A job completes "successfully" with a `result.txt` placeholder instead
of an actual image/mesh/audio file. The user has no way to know they got a stub.

**Impact**: Users may think inference is working when it's not.

**Tasks**:
- [ ] Add a `"stub": true` field to job output metadata when fallback fires
- [ ] Client should display a warning: "This result is a placeholder — ML dependencies not installed"
- [ ] Add `assgen server status --check-inference` that verifies key imports are available
- [ ] Surface missing dependency info in `GET /health` response

### GAP-10: No Plugin System for Game Engines [LOW for MVP]

**Problem**: README mentions Godot/Unreal/Unity plugins. None exist. Engine integration
is handled via the `pipeline.integrate.export` handler which converts files to engine
formats but doesn't provide in-editor UI.

**Impact**: Low for MVP — the CLI + file export workflow is sufficient. Plugins are a
post-MVP feature.

**Tasks** (post-MVP):
- [ ] Godot GDScript addon: HTTP client -> assgen server, import result into scene
- [ ] Unity Editor window: same pattern in C#
- [ ] Unreal Editor plugin: same pattern in C++

### GAP-11: Single Worker Thread [LOW for MVP]

**Problem**: The server runs a single `WorkerThread` that processes one job at a time.
GPU memory constraints make this reasonable for inference, but it means compose pipelines
(12 steps) monopolize the queue.

**Impact**: Acceptable for single-user MVP. Would need attention for multi-user.

**Tasks** (post-MVP):
- [ ] Consider separating CPU-only handlers (mesh ops, validation) into a second worker
- [ ] Add job priority preemption for quick utility tasks

## 5. MVP Definition

**MVP Goal**: A single developer on a laptop can generate game assets (concept art, 3D
meshes, textures, audio, and multi-step NPC pipelines) via a self-hosted GPU server on
their home LAN.

### MVP Acceptance Criteria

1. `assgen-server start` runs on Windows desktop with CUDA and accepts remote connections
2. `assgen client config set-server http://<desktop-ip>:8432` on the laptop works
3. `assgen gen visual concept generate --prompt "medieval sword" --wait` produces a real image
4. `assgen gen audio sfx generate "laser blast" --wait` produces a real WAV file
5. `assgen compose npc "pig shopkeeper" --wait` completes all steps end-to-end
6. `assgen jobs list --json` returns valid JSON
7. Stub fallback is clearly signaled to the user
8. Setup takes < 30 minutes following documentation

3. Implement `run()` function with the signature above
4. Run `pytest tests/test_catalog.py` to verify importability
